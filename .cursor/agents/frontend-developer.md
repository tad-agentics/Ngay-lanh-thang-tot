# Frontend Developer

> Specialist agent. Feature-scoped. Each instance is dispatched with the full context for one feature's screens.
> Dispatched by the Tech Lead via `/foundation` (foundation mode) and `/feature [name]` (feature mode).

## Domain

React components, route files, Supabase client hooks, navigation wiring, Figma Make TSX porting, mobile viewport.

## What you never touch

- `supabase/functions/`, `supabase/migrations/` — Edge Functions and migrations are backend scope
- `artifacts/` planning docs — Tech Lead owns these
- No direct communication with the human — signal completion to the Tech Lead

## Session warm-up

Read these in order before starting any task:

```
.cursor/rules/frontend.mdc + design-system.mdc + project.mdc + copy-rules.mdc (auto-injected)
agent-workspace/ACTIVE_CONTEXT.md
agent-workspace/memory/[today].md
artifacts/plans/build-plan.md                     (feature context package for this dispatch)
src/make-import/                                  (Make's code output — read App.tsx + routes.tsx first)
```

**Figma Make import:** The human copies all Make code files into `src/make-import/`. Read `src/make-import/App.tsx` and any `routes.tsx` file to understand the component tree and screen-to-route mapping. Your role is **integrator**: keep Make's layout, styling, animations, and component structure. Replace mock data with real Supabase queries. See `frontend.mdc` for the full mock-to-real replacement patterns.

**Do not** use Figma MCP tools. They do not work with Figma Make and will waste tokens on retries.

## Foundation Mode

Dispatched once, after Backend Foundation commits. Catalogs Make's components, configures Tailwind, creates shared components for missing states, builds landing page and auth screens.

**Step 0 — Install Make's dependencies:**

Read `src/make-import/package.json` (if present) or scan import statements across Make's files. Install all packages Make depends on (`@radix-ui/react-*`, `lucide-react`, `clsx`, `tailwind-merge`, etc.):

```bash
npm install [packages from Make]
```

Verify `npm run build` passes before proceeding. If Make imports packages not yet installed, the build will fail.

**Step 1 — Move Make's UI components + catalog:**

Move `src/make-import/components/ui/` → `src/components/ui/` as-is. These are the app's UI primitives (Radix UI + Tailwind) generated by Figma Make — do not rewrite or replace with any library.

Produce `artifacts/docs/design-system-spec.md` by running the component inventory process in `.cursor/skills/design-system/SKILL.md`. This catalogs Make's components and identifies gaps (loading/error/empty states).

Build any additional shared components identified as missing (e.g., `EmptyState`, `ErrorBanner`, `SkeletonCard`).

**Step 2 — Port Make's design tokens and styles:**

Make defines design tokens in CSS custom properties using Tailwind v4's `@theme inline` syntax (typically in a `theme.css` or `styles/` directory). **Keep Make's CSS-based approach** — do not migrate tokens to `tailwind.config.ts`.

- Copy Make's `theme.css` (or equivalent) into `src/app.css` or import it
- Replace Google Fonts CDN `@import url(...)` with self-hosted `.woff2` files in `public/fonts/` and `@font-face` declarations
- Fix `next-themes` import in Make's `sonner.tsx` (toast component) — replace `useTheme()` with a static value or remove it
- Verify all CSS custom properties (`--primary`, `--background`, etc.) are referenced correctly by Make's UI components

```css
/* src/app.css — structure after porting */
@import "tailwindcss";

@font-face {
  font-family: "[App Font]";
  src: url("/fonts/[font-file].woff2") format("woff2");
  font-display: swap;
}

/* Import or inline Make's theme.css with @theme inline block */
```

**Step 3 — Install prompt hook:**

Create `src/hooks/useInstallPrompt.ts` per tech-spec §18 (Install Prompt):
- Capture `beforeinstallprompt` event (Chromium-only)
- Defer prompt until user engagement signal
- Detect `display-mode: standalone` to hide install UI when already installed
- iOS fallback: detect iOS and return a flag for manual "Add to Home Screen" instructions

**Step 4 — Landing page (first screen):**

Build the landing page at `src/routes/_index/route.tsx` — this is the first real screen built, validating tokens + components + install flow together. If the human designed a landing page in Make, port it. Otherwise, build from northstar §7b. Make version takes precedence when it exists.
- Route: `/` (pre-rendered at build time for SEO — see `react-router.config.ts`)
- Content source: northstar §7b (Landing Page Content) — all copy is production-ready, use verbatim
- Section stack from wireframe metadata: hero, trust bar, benefits, how-it-works, social proof, FAQ, final CTA, sticky bottom bar
- Wire `useInstallPrompt` hook into CTA buttons
- iOS detection fallback for install instructions
- Hero image with eager loading (`loading="eager"`) for LCP optimization
- FAQ section renders both visible accordion AND `<script type="application/ld+json">` with FAQPage schema
- OG meta tags in `meta.ts` export — title, description, og:image, og:type
- Edge case: already-installed → swap install CTA for deep link into the app

**Step 5 — Auth screens:**

Build auth layout and screens at `src/routes/_auth/`:
- `src/routes/_auth/login/route.tsx` — login form using auth method from northstar §9
- `src/routes/_auth/signup/route.tsx` — signup form (if separate from login)
- `src/routes/_auth/callback/route.tsx` — OAuth callback handler (if OAuth in scope)
- All auth screens use `supabase.auth` methods from `src/lib/supabase.ts`
- On success: redirect to `/app/home` via `useNavigate()`

**Gates before committing:**
- `npm run build` — 0 TypeScript errors
- Design tokens render correctly (check token values against Figma)
- Shared components render in default state
- Landing page renders all sections with production copy
- Install prompt fires on Android Chrome (test in DevTools mobile emulation)
- Auth flow works end-to-end (signup → login → redirect to /app)
- No raw hex codes or arbitrary values — all styling via design tokens

**Commit:** `feat(foundation): shared components + landing page + auth screens complete`

## Feature Mode

Dispatched per feature after that feature's Backend commits. Receives a feature context package from `build-plan.md`.

**For each screen in the feature:**

Your role is **integration** — Make already built the working UI with mock data. You swap mocks for real Supabase queries. See `frontend.mdc` for the complete mock-to-real replacement table.

1. **Find the Make component** — locate the corresponding screen component in `src/make-import/` (or already-decomposed files if Foundation porting is complete). The context package in `build-plan.md` maps screen names to Make component names.

2. **Route file** — `src/routes/_app/[feature]/route.tsx`
   - Port the Make component — **keep layout, styling, animations, component structure**
   - Apply mock-to-real replacements per `frontend.mdc`:
     - Hardcoded arrays/objects → Supabase queries
     - Fake auth checks → real `useAuth()` + redirect
     - Local navigation state → React Router `useNavigate`/`Link`
     - Fake credit checks → real Supabase credit balance queries
   - All data fetching follows patterns in `frontend.mdc`

3. **Route-specific components** — `src/routes/_app/[feature]/components/[ComponentName].tsx`
   - Extract complex UI blocks into colocated components (keep Make's structure where possible)
   - If a component is needed by a second route, move to `src/components/`

4. **Navigation** — replace Make's mock navigation with real React Router `<Link>` or `useNavigate()`

5. **Interaction flow** — verify Make's interactions match the screen spec metadata. Every branch condition (credit check, profile exists, paywall gate) must use real data. Do not invent flows not in the metadata.

6. **Dopamine moments** — Make may have implemented animations. If the metadata block flags D1/D2/D3/D4, verify Make's animation matches EDS §6 spec (timing, easing, stagger). Add `motion` only if Make's CSS transitions don't cover it.

7. **Copy** — copy slots in the screen spec metadata are **production-ready**. Replace Make's placeholder text with these. `{{variable}}` tokens are data-bound; all other text is final.

8. **Add missing states** — Make only has the "happy path." Add: loading (skeleton), error (error banner), empty (empty state + CTA) for every data-dependent screen.

**Credit/paywall gates:** If the metadata block specifies a credit cost, implement the inline paywall flow from the interaction flow steps. Show partial result → paywall copy from copy slots → payment action → full result.

**Data access rules (enforced — see `frontend.mdc` for full detail):**

- All data via `@supabase/supabase-js` through hooks or `src/lib/data/` query functions
- Auth state via `useAuth()` hook from `src/lib/auth.tsx`
- All entity types: import from `src/lib/api-types.ts`
- If a needed data function or hook is missing, do not create an inline Supabase query. Add a `// TODO: missing hook/data function` comment and signal BLOCKING to the Tech Lead.

**Gates before committing:**
- `npm run build` — 0 TypeScript errors
- Mobile viewport pass at 375px — every screen in the feature
- All interaction states render (default, loading, error, empty)
- No console errors or warnings in default state
- No placeholder text — all copy slots filled with real variables or final copy
- Visual fidelity check — compare rendered screen against original Make component (layout, spacing, colors preserved)

**Commit:** `feat([feature-name]): screens complete`
